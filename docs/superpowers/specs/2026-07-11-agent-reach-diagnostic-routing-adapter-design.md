# Agent-Reach診断連動アダプター設計

## 目的

TOMOSのWeb調査で、Agent-Reachが診断した利用可能な上流ツールをチャネル別に選択する。
取得結果は既存のTOMOS共通形式へ正規化し、出典照合、一覧回答、キャラクター口調、表示処理は既存TOMOSが担当する。

## 背景

現在のTOMOSはAgent-Reachを導入・診断できるが、調査実行時はJina Reader、`yt-dlp`、`gh`、RSS処理などを固定的に呼び出している。
診断結果が実際の経路選択に反映されないため、Agent-Reachが推奨経路を変更してもTOMOSは追従できない。

Agent-Reachは単一の検索APIではなく、チャネルごとに利用可能な上流ツールを診断し、AIエージェントへ経路を示す能力レイヤーである。
TOMOSではこの役割を診断連動アダプターとして接続する。

## 初回対象

- Webページ本文取得
- 全Web検索
- YouTube字幕取得
- GitHub公開情報の取得と検索
- RSS/Atom取得

## 対象外

- Twitter、Reddit、小紅書、Facebook、InstagramなどのSNS
- Cookie、ログイン情報、APIキーの保存
- Agent-Reach本体コードの変更
- 外部調査結果の長期記憶への自動保存
- 投稿、コメント、Issue作成などの書き込み操作
- Agent-Reachが未導入の環境での自動インストール

## 実装前提

現在のルート`AGENTS.md`はInternet LayerとWeb、GitHub、YouTube、RSS連携処理の変更を禁止している。
コード実装は、プロジェクト管理者がこの作業を許可する規則へ正式に更新した後に開始する。
設計書と実装計画の作成だけでは、この禁止規則を解除したものと扱わない。

## 採用方式

TOMOS側に診断連動アダプターを追加する。
Agent-Reach本体をライブラリとして改変せず、`agent-reach doctor`の結果と許可済み上流ツールを使う。

### 代替案との比較

1. 診断連動アダプター
   - Agent-Reachの経路変更へ追従できる。
   - TOMOSの既存回答処理と安全制限を維持できる。
   - 初回採用とする。
2. Agent-Reachを別のAIエージェントとして起動
   - 自由度は高いが、出力形式、権限、停止条件が不安定になる。
   - 採用しない。
3. 現状の固定経路を維持
   - 実装は少ないが、診断結果を使わない問題が残る。
   - 採用しない。

## 構成

### AgentReachDoctorCache

- `agent-reach doctor`の結果を取得する。
- 有効期限は5分とする。
- 手動診断後はキャッシュを更新する。
- 失敗時は直前の有効な診断結果を使い、存在しなければ現行TOMOS経路へ戻る。

### AgentReachRouteSelector

入力:

- チャネル
- ユーザー入力
- 診断結果

出力:

- 選択したバックエンド名
- 実行種別
- フォールバック候補
- 選択理由

初回の経路:

| 用途 | 優先経路 | フォールバック |
| --- | --- | --- |
| Webページ本文 | Jina Reader | 現行TOMOS本文取得 |
| 全Web検索 | Exaが利用可能ならExa | 現行`search_web` |
| YouTube字幕 | `yt-dlp` | 現行TOMOS字幕取得 |
| GitHub | `gh` | 現行TOMOS GitHub処理 |
| RSS/Atom | `feedparser` | 現行TOMOS RSS処理 |

### AgentReachExecutor

- シェル文字列を実行しない。
- 許可リストに登録した実行ファイルと固定引数だけを配列形式で実行する。
- チャネル別にタイムアウトと最大出力サイズを設定する。
- 標準エラーをユーザー回答へ直接混ぜない。
- 認証を要求する経路は初回対象から除外する。

### AgentReachResultNormalizer

すべての取得結果を以下へ変換する。

```json
{
  "title": "string",
  "url": "https://example.com/",
  "snippet": "string",
  "source": "agent-reach:web",
  "backend": "jina",
  "routeReason": "doctorで利用可能と確認"
}
```

既存の検索結果、出典表示、一覧根拠処理がこの形式をそのまま利用する。

## データフロー

1. ユーザーがWeb調査を有効にして送信する。
2. TOMOSが質問から必要なチャネルを判定する。
3. 診断キャッシュから利用可能なバックエンドを取得する。
4. 経路選択器が優先経路とフォールバックを決める。
5. 実行器が許可済み上流ツールを一度実行する。
6. 失敗した場合だけ現行TOMOS経路を一度実行する。
7. 正規化した結果を既存の出典照合と回答生成へ渡す。
8. 回答下の診断情報へ使用経路、成功状態、フォールバック理由を表示する。

## 安全性

- ローカルIP、リンクローカル、予約IPへの外部URLアクセスを拒否する。
- URLのユーザー名、パスワード、規定外ポートを拒否する。
- リダイレクト先も同じURL検証を通す。
- コマンド名と引数は許可リストから構築する。
- `shell=True`を使わない。
- Cookie、Token、環境変数の値をログへ出さない。
- 調査結果を長期記憶へ自動保存しない。
- 同じチャネルで無限再試行しない。

## エラー表示

- 優先経路成功: 使用経路だけを表示する。
- 優先経路失敗、フォールバック成功: 使用経路とフォールバック理由を表示する。
- 両方失敗: 取得できなかった理由と成功条件を表示する。
- 上流ツールの生エラーは内部ログへ保持し、画面では個人情報やパスを除いた短い日本語へ変換する。

## UI

既存の「Web調査」操作を維持する。
初回ではチャネル選択UIを増やさない。
回答下の診断情報に以下を追加する。

- 使用経路
- 取得状態
- フォールバックの有無
- 失敗時の確認方法

## テスト

### 単体テスト

- 診断結果から4チャネルの優先経路を選べる。
- 診断結果が期限内なら`doctor`を再実行しない。
- 未対応バックエンドを選択しない。
- 優先経路失敗時にフォールバックを一度だけ実行する。
- ローカルIPと規定外コマンドを拒否する。
- 上流結果をTOMOS共通形式へ正規化する。

### 回帰テスト

- Agent-Reach未導入でも現在のWeb調査が動く。
- YouTube字幕成功時の回答品質を維持する。
- GitHub URLの取得結果を維持する。
- RSS URLの取得結果を維持する。
- 一覧回答の出典照合と架空項目除外を維持する。
- Web調査結果を長期記憶へ自動保存しない。

### 手動確認

- 一般Webページの要約
- 公式サイトの一覧
- 字幕付きYouTube動画の要約
- 公開GitHubリポジトリの説明
- RSSフィードの記事一覧
- Agent-Reach停止時のフォールバック

## 受け入れ条件

- 回答ごとに実際に使用したバックエンドが確認できる。
- `doctor`で利用不可のバックエンドを実行しない。
- 優先経路失敗時も既存TOMOS経路で回答を継続できる。
- 出典にない固有名詞、数値、項目を追加しない。
- Agent-Reach未導入環境で既存機能を壊さない。
- SNS、Cookie、ログイン情報へアクセスしない。
- 全関連テストと`git diff --check`が通る。
