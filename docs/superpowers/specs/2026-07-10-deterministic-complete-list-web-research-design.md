# Web調査の決定論的一覧回答 設計仕様

## 1. 背景

TOMOSのWeb調査で「全シリーズ」「すべて」「一覧」「箇条書き」などを依頼すると、取得したページ本文から候補を抽出した後、ローカルAIが一覧本文を再生成している。現在は生成後に正規表現で未確認項目を除去・分類しているが、次の問題が繰り返し発生している。

- 出典にない名称をローカルAIが追加する。
- 出典にある名称をローカルAIが省略する。
- ページ内リンクの追跡で、ログイン、アカウント作成、画像、無関係な解説ページまで取得する。
- 生成後の正規表現では、誤った名称、欠落、分類崩れを完全には直せない。
- ストリーミング中は、後段で修正する前の誤った一覧が画面に表示される。
- 合成データの単体テストは通っても、実際のWebページ構造では同じ品質にならない。

これは個別の抽出条件ではなく、事実一覧を生成AIに再生成させる構造が根本原因である。

## 2. 目的

Web調査の一覧回答では、取得した根拠から確認できた項目だけをTOMOS側で決定論的に表示する。キャラクター性は維持するが、ローカルAIに事実項目の追加、削除、改名、並べ替えをさせない。

## 3. 対象範囲

### 対象

- 「全シリーズ」「すべて」「一覧」「箇条書き」「リスト」「網羅」などの一覧依頼
- Web調査で取得したページ本文を根拠にする通常チャット回答
- ストリーミング回答と非ストリーミング回答
- 回答に表示するWeb調査の出典

### 対象外

- Agent-Reach本体の変更
- Web検索、YouTube字幕、GitHub、RSSの取得方式そのものの変更
- 特定作品、特定企業、特定サイト専用の名称辞書
- Web調査結果の長期記憶への保存
- クラウド同期、外部API、追加依存

## 4. 検討した方式

### A. 決定論的一覧本文＋キャラクター導入文

TOMOSが根拠ページから候補を抽出し、順序と表記を保持したまま一覧本文を組み立てる。ローカルAIは一文の導入文だけを生成する。

- 長所: 架空項目を構造的に防げる。結果を再現しやすい。既存のローカル構成を維持できる。
- 短所: 複雑な表や特殊なページでは、抽出できる件数が少なくなる。

### B. ローカルAIによる抽出と別推論による検証

1回目の推論で一覧を抽出し、2回目の推論で根拠との一致を確認する。

- 長所: 複雑な文章からも柔軟に抽出できる。
- 短所: 遅い。同じ入力でも結果が変わる。2回目の推論も誤る可能性がある。

### C. サイト別の専用パーサー

公式サイトやWikipediaなど、サイトごとにHTML構造を解釈する。

- 長所: 対応済みサイトでは高精度に抽出できる。
- 短所: 汎用性が低く、サイト更新のたびに保守が必要になる。

### 採用方式

Aを採用する。BとCは初期実装に含めない。特定サイトで継続的な需要が確認された場合だけ、Aの候補抽出器を差し替える拡張としてCを検討する。

## 5. 設計方針

### 5.1 責務分離

一覧回答を次の3段階に分離する。

1. 根拠選択: 信頼度スコアを使い、一覧の根拠とする単一ドメインを選ぶ。
2. 候補抽出: 選んだ根拠ページから、表示可能な一覧項目を抽出する。
3. 回答生成: TOMOSが候補を決定論的にMarkdownへ変換し、キャラクターの導入文を付ける。

ローカルAIが生成した箇条書き、見出し、件数、作品名、日付、分類は一覧本文として採用しない。

### 5.2 データ型

`server.py` 内に、一覧回答専用の不変データを追加する。

```python
@dataclass(frozen=True)
class CompleteListEvidence:
    query: str
    source_domain: str
    source_results: tuple[dict[str, str], ...]
    candidates: tuple[str, ...]
    status: str
    warnings: tuple[str, ...]
```

`status` は次の3値に固定する。

- `source-backed`: ページ本文から3件以上の項目を抽出できた。
- `partial`: 根拠ページは取得できたが、抽出できた項目が1〜2件だった。
- `unavailable`: 一覧の根拠にできるページ本文を取得できなかった。

`source-backed` は「すべて取得できた」という意味ではない。完全性を断定する文言は、根拠ページ自身に明示的な全件表示の情報がある場合でも初期実装では使用しない。

### 5.3 公開インターフェース

```python
def build_complete_list_evidence(
    query: str,
    results: list[dict[str, str]],
) -> CompleteListEvidence:
    ...


def render_complete_list_answer(
    character_intro: str,
    evidence: CompleteListEvidence,
) -> str:
    ...


def public_search_results_for_answer(
    results: list[dict[str, str]],
    evidence: CompleteListEvidence | None,
) -> list[dict[str, str]]:
    ...
```

各関数の責務は次のとおりとする。

- `build_complete_list_evidence`: 根拠ドメインの選択、候補抽出、重複除去、状態判定を行う。
- `render_complete_list_answer`: 候補を変更せず、回答本文を組み立てる。
- `public_search_results_for_answer`: 一覧回答に実際に使用した根拠だけを画面へ返す。

## 6. データ処理

### 6.1 根拠選択

- `Webページ本文:` または `agent-reach:web` の結果だけを根拠候補にする。
- 公式・公共機関を第1群、百科事典・専門Wikiを第2群、その他を第3群として信頼度を分ける。
- 第1群と第2群に候補がある場合は、抽出できた有効候補数が最も多い単一ドメインを選ぶ。同数の場合だけ信頼度が高い群を優先する。
- 第1群と第2群から有効候補を抽出できない場合だけ、第3群を候補にする。
- 選択後は別ドメインの候補を混ぜない。
- 一覧回答のためにページ内の任意リンクを再帰的に追跡しない。
- 検索結果一覧、ログイン、アカウント作成、画像URL、URL解説などは根拠から除外する。
- 生の検索結果は内部診断に残せるが、回答の出典欄には表示しない。

### 6.2 候補抽出

- 選択した根拠ページの本文だけから抽出する。
- Markdownの箇条書き、表セル、人が読めるリンクラベルを構造化候補として扱い、説明文の平文から名称を推測しない。
- 候補文字列が根拠本文にそのまま存在することを必須とする。
- 元ページでの出現順を維持する。
- 正規化した文字列が同じ候補だけを重複として除外する。
- 候補の表記をローカルAIで修正しない。
- 見出し、日時、画像名、URL、説明文、メタデータ、ナビゲーション文言は除外する。
- 分類に確信がない候補を推測で分類しない。初期実装は原則として単一一覧で表示する。
- 最大表示件数は100件とし、超過時は「100件まで表示」と明記する。

### 6.3 キャラクター導入文

- ローカルAIには、ユーザーの呼び方とキャラクター口調を反映した一文だけを生成させる。
- 生成文は最大2行とし、箇条書き、見出し、固有名詞、件数、日付、数値を含む場合は破棄する。
- 破棄した場合は、キャラクター設定から生成済みの安全な定型文を使う。
- 初期の定型文は「確認できた内容をまとめたよ。」とする。
- VOICE.mdのキャラクター安定化を維持し、正確性より口調を優先しない。

### 6.4 回答本文

`source-backed` の回答は次の形式に固定する。

```markdown
{キャラクター導入文}

## 確認できた項目（{件数}件）
- {候補1}
- {候補2}

## 確認できていない点
- 取得した根拠ページで確認できた項目だけを掲載しています。
```

`partial` の場合は取得できた候補を表示し、「完全な一覧としては確認できませんでした」と明記する。`unavailable` の場合は一覧を生成せず、根拠ページ本文を取得できなかった理由と再試行方法を表示する。

### 6.5 ストリーミング

- 一覧回答はローカルAIの生の生成チャンクを画面へ送らない。
- `start` イベントは従来どおり送る。
- 導入文の生成と決定論的一覧の組み立てが完了してから、完成済み本文を1回の `chunk` と `done` で送る。
- これにより、後段修正前の架空項目が一瞬表示される問題を防ぐ。

### 6.6 出典表示

- `search.results` には `CompleteListEvidence.source_results` だけを返す。
- 同じURLは1回だけ表示する。
- 候補抽出に使っていない検索結果は、通常の出典欄に表示しない。
- 内部診断には、選択ドメイン、根拠ページ数、抽出候補数、状態を保存する。
- ページ本文全文は診断や長期記憶へ保存しない。

## 7. 診断情報

一覧回答時は、既存のWeb調査診断へ次を追加する。

```json
{
  "type": "complete-list-grounding",
  "status": "success|warning|error",
  "label": "一覧根拠",
  "message": "単一の根拠ページから18件を確認しました。",
  "sourceDomain": "example.com",
  "sourceCount": 1,
  "candidateCount": 18,
  "mode": "deterministic-complete-list"
}
```

候補数が0件の場合は、エラー理由をユーザー向けに短く表示する。内部例外やページ本文は表示しない。

## 8. エラー処理

- 根拠ページ取得失敗: 検索結果のタイトルだけで一覧を生成しない。
- 候補0件: 「一覧項目を抽出できませんでした」と表示し、再検索または対象ページURLの指定を案内する。
- 候補1〜2件: `partial` として表示し、完全な一覧と断定しない。
- 候補100件超: 先頭100件を元ページ順で表示し、表示上限を明記する。
- ローカルAI失敗: 安全な定型導入文を使い、決定論的一覧は表示する。
- 出典URL不正: 該当結果を除外し、残る根拠で処理する。

## 9. プライバシーと保存

- Web調査結果、候補一覧、ページ本文は長期記憶へ自動保存しない。
- チャット履歴への保存は既存仕様を変更しない。
- APIキー、Cookie、認証情報、ローカルファイル本文を診断へ含めない。
- 外部APIやクラウドサービスを追加しない。

## 10. テスト方針

実装はテスト駆動で進める。最初に失敗する回帰テストを追加し、その後に最小実装を行う。

### 単体テスト

- 単一ドメインだけが根拠として選ばれる。
- 信頼できる複数ドメインでは、有効候補数が多い出典が選ばれる。
- 有効候補数が同じ場合は、公式性が高い出典が選ばれる。
- 無関係なWikipediaナビゲーションページが除外される。
- 候補の順序と表記が維持される。
- 同じ候補だけが重複除去される。
- AI生成本文に架空項目があっても回答へ入らない。
- AI生成本文が途中で切れても、一覧本文は欠落しない。
- 導入文に固有名詞や件数が含まれる場合は定型文へ置換される。
- `partial` と `unavailable` の文言が完全性を断定しない。
- 公開出典が実際の根拠ページだけになる。

### APIテスト

- ストリーミング時にAIの生チャンクが送られない。
- 完成済み本文が1回の `chunk` と `done` で一致する。
- 非ストリーミングでも同じ本文と出典が返る。
- 診断に `deterministic-complete-list` と候補数が含まれる。

### 回帰テスト

- 一覧以外の通常Web調査回答は従来どおりローカルAIが生成する。
- YouTube字幕分析、GitHub調査、RSS調査に影響しない。
- Web調査を使わない通常チャットに影響しない。
- キャラクターの呼び方と一人称が維持される。

## 11. 成功条件

- 同じ根拠入力から毎回同じ一覧本文が生成される。
- ローカルAIが生成した架空項目は回答に1件も表示されない。
- 根拠から抽出済みの候補が、ローカルAIの出力長によって欠落しない。
- 回答の出典欄に、一覧作成で使用していないページが表示されない。
- ストリーミング中を含め、修正前の一覧が画面に表示されない。
- 一覧回答でもキャラクターの短い導入文が維持される。
- 既存の通常チャット、動画分析、教材パック、長期記憶に回帰がない。

## 12. 導入とロールバック

- 初期導入は既存の一覧判定条件にだけ適用する。
- 一覧以外のWeb調査経路は変更しない。
- 問題が出た場合は、一覧回答専用の決定論的レンダラー呼び出しを外し、既存経路へ戻せる構造にする。
- 旧来の生成後フィルターは、一覧回答では呼ばず、実装安定後に別コミットで削除を検討する。
