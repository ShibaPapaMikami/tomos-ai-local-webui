# note記事作成サポート 配布・インストール設計

## 目的

TOMOSの教材パック画面に「note記事作成サポート」を常設し、利用者が外部教材のフォルダー構成を理解しなくても、画面からダウンロードして利用開始できるようにする。

公開GitHub Releasesを配布元として利用し、専用の配布サーバーやAPIキーを必要としない構成にする。

## 対象範囲

- 教材パック画面の「note記事作成サポート」カード
- GitHub Releases上のZIP取得
- ZIPの安全性・整合性検証
- ローカルへのインストール、更新、削除
- 起動時の導入状態復元
- 配布用ZIPとSHA-256の生成手順

## 対象外

- noteへのログイン、投稿、下書き保存
- 自動ダウンロード、自動更新
- 任意URLや第三者リポジトリからの導入
- GitHub認証やAPIキーの保存
- Agent-Reach本体の変更
- 有料の配布サーバー

## 利用者向け導線

教材パック画面に、次の固定カードを表示する。

- 名称: `note記事作成サポート`
- 説明: `note向けの記事構成、本文整理、公開前確認を支援します。`
- 状態: `未導入`、`ダウンロード中`、`利用可能`、`更新あり`、`失敗`
- 操作: `ダウンロードして追加`、`更新`、`削除`

未導入時のボタンを押すと、GitHubからファイルを取得することを確認する。利用者が承認した場合だけ通信を開始する。

既存の「外部教材データをインポート」は、独自教材を追加する上級者向け機能として詳細表示へ移す。note記事作成サポートの通常導線では使用しない。

## 配布方式

GitHub Releasesに、バージョン付きZIPを添付する。

例:

```text
リリースタグ: note-article-writing-v0.1.0
アセット: TOMOS-note-article-writing-v0.1.0.zip
```

アプリ内には次の許可済み情報を静的に保持する。

- パックID: `note-article-writing`
- 表示名
- バージョン
- GitHubリポジトリ
- リリースアセットのHTTPS URL
- アセット名
- SHA-256

利用者が任意URLを入力する機能は設けない。配布元の変更はTOMOSの更新として管理する。

## パック構成

ZIP直下にパック本体のフォルダーを1つ置く。

```text
note-article-writing/
  pack.json
  README.md
  prompts/
    outline.txt
    draft.txt
    polish.txt
    publish-check.txt
```

`pack.json`には少なくとも次を含める。

```json
{
  "id": "note-article-writing",
  "name": "note記事作成サポート",
  "version": "0.1.0",
  "prompts": {
    "outline": "prompts/outline.txt",
    "draft": "prompts/draft.txt",
    "polish": "prompts/polish.txt",
    "publishCheck": "prompts/publish-check.txt"
  }
}
```

## サーバー責務

ブラウザー側へZIP展開依存を追加せず、Pythonサーバーが標準ライブラリで取得・検証・展開する。

想定API:

- `GET /api/study-packs/catalog`: 許可済みパックと導入状態を返す
- `POST /api/study-packs/note-article/install`: 新規導入または更新する
- `POST /api/study-packs/note-article/remove`: 対象パックだけ削除する

保存先:

```text
.gemma4-data/study-packs/note-article-writing/
```

ダウンロード後は一時フォルダーで検証し、成功時だけ保存先を置き換える。更新失敗時は導入済みバージョンを維持する。

## 検証条件

- HTTPSかつ許可済みGitHub URLである
- 応答サイズが上限以内である
- SHA-256がアプリ内定義と一致する
- ZIP内に絶対パス、`..`、シンボリックリンクがない
- `pack.json`が存在する
- パックIDとバージョンが期待値に一致する
- 4種類のプロンプトが存在し、読み取り可能である

不一致の場合は保存先を変更せず、日本語のエラーを画面へ返す。

## フロントエンド責務

画面を開いた時にカタログAPIを読み、導入状態をカードへ反映する。導入済みなら既存の教材パック選択処理へ同じ定義を登録し、4種類の作成モードを利用可能にする。

ローカルファイルを導入状態の正本とする。ブラウザーのローカル保存は表示・選択状態のキャッシュとして扱い、消去されても次回読込時に復元する。

削除時は、対象パックのローカルファイル、登録定義、選択状態だけを削除する。学習セット、他の教材パック、会話履歴は変更しない。

## エラー表示

内部例外をそのまま表示せず、原因別に短く案内する。

- `ダウンロードできませんでした。インターネット接続を確認してください。`
- `配布ファイルを確認できませんでした。TOMOSを更新してから再度お試しください。`
- `インストールできませんでした。現在の教材パックは変更されていません。`

## 配布物の作成

再現可能なZIPとSHA-256を生成するスクリプトをリポジトリへ追加する。ZIP内のファイル順、時刻、改行を固定し、同じ内容から同じハッシュを生成できるようにする。

GitHub Releaseの作成とアセット公開は外部への変更にあたるため、実装・ローカル検証後に利用者の明示確認を受けて実行する。

## テスト

- カードの5状態と操作ボタン
- 確認前に通信しないこと
- 成功時の取得、検証、展開、再読込
- SHA-256不一致、サイズ超過、欠損ファイル
- ZIP内のパストラバーサルとシンボリックリンク拒否
- 誤ったパックID・バージョンの拒否
- 更新失敗時の旧バージョン維持
- 削除時に学習セットと他パックを維持
- GitHubへ接続できない場合の日本語表示
- 実通信を行わないモックテスト
- UI変更時のPWAキャッシュ更新

## 実装順序

1. 配布カタログとパック検証処理をテストから追加する
2. インストール・更新・削除APIを追加する
3. 教材パック画面へ固定カードを追加する
4. 起動時の復元と既存選択処理へ接続する
5. 配布ZIP生成スクリプトを追加する
6. ローカルの模擬配布物で一連の動作を確認する
7. 明示確認後にGitHub Releaseへ公開する

## 完了条件

- 未導入環境で、確認操作後に1クリックで導入できる
- 再起動後も利用可能状態と4種類のモードが復元される
- 更新・削除が他データへ影響しない
- 破損・改ざん・危険なZIPを導入しない
- GitHub未接続でも既存TOMOS機能が継続する
